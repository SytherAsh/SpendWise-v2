# Epic 4 Handoff

Written 2026-07-02 at the end of the ML data-labeling session, for whoever starts the
Epic 4 (ML Categorization Service) implementation session next. Read this first, then
`implementation/epics/epic-04-ml-categorization.md` for the actual task breakdown.

## Dataset: cleaned, labeled, finalized

- `ml/data/spendwise_labeled.xlsx` — **1,810 labeled transactions**, ready for
  `ml/training/train.py`. Gitignored (real personal data), same as the raw source.
- Produced from `spendwise_4yr_sbi_statement` (1,920 raw rows) via the pipeline in
  `ml/labeling/` — rule-based auto-labeling (`merchant_rules.csv`, ~400+ rules) plus an
  interactive interview for the long tail. Full provenance in
  `ml/labeling/tracking/LABELING_STATUS.md`.
- **110 rows / 59 recipient groups intentionally excluded**, not silently dropped —
  mostly payment-rail intermediaries (Google Pay, Paytm, Mobikwik, Razorpay) where the
  real recipient is masked, plus a few unresolved names. Still sitting in
  `ml/labeling/datasets/spendwise_4yr_sbi_statement/review_sheet.xlsx` if anyone wants
  to extend coverage later — re-running the pipeline picks up where it left off.
- Two known data-quality bugs in the raw source were found and fixed at the pipeline
  level (not just worked around for this one dataset): placeholder `transaction_id`
  reuse across genuinely different transactions (fixed via content-hash synthesis in
  `merge_datasets.py`), and a couple of manually-patched stipend-income rows that
  couldn't be matched by the normal recipient-name rule mechanism (documented in
  `LABELING_STATUS.md`).

## Category set: 12, not 10 — this changed mid-session

`Medical` (id 11) and `Fees & Debt` (id 12) were added because medical-shop and
fee/debt transactions kept surfacing with no home among the original 10 and were being
forced into Miscellaneous. This touched more than docs:

- `docs/requirements.md` — canonical category list, now 12
- `backend/src/main/resources/db/migration/V7__add_medical_and_fees_categories.sql` —
  new migration (additive; `V2` couldn't be edited, it's already Flyway-checksummed).
  **Not yet applied to any running database** — just committed as a file.
- `docs/database.md`, `docs/api.md` — updated seed data / endpoint description
- `ml/labeling/scripts/categories.py` — script source of truth, now 12
- Retroactively recategorized 6 rows that were labeled before the category existed

**Category distribution is uneven** — Transfers is ~50% (897/1,810, expected — P2P
transfers are naturally frequent). **Sports & Fitness has zero examples in this
dataset; Fees & Debt has only 2.** Expect to need class weighting, and don't be
surprised if the model can't predict Sports & Fitness at all from this data alone. Full
table in `ml/labeling/tracking/LABELING_STATUS.md`.

## Two separate knowledge bases — do not confuse them

During labeling, a lot of "who is this" context came up (friend, family, self-transfer,
merchant, employer, subscription service) alongside the category work. That's genuinely
useful for a future Analytics feature (breaking Transfers down by who it went to), but
**it is explicitly not part of Epic 4 and never should become additional ML
categories.** See ADR-010 in `docs/decisions.md` for why, and `docs/architecture.md`
"Future Enhancement: Counterparty Metadata Enrichment" for the proposed (unbuilt) design.
Two artifacts now exist, deliberately kept separate:

1. **Category Knowledge Base — complete, and this is what Epic 4 trains on.**
   `ml/labeling/knowledge_base/merchant_rules.csv` + `keyword_rules.csv` (~400 rules),
   feeding `ml/data/spendwise_labeled.xlsx`'s `category` column — exactly the 12 values
   in `docs/requirements.md`, nothing else. This is what `auto_label.py` /
   `apply_review.py` / `merge_datasets.py` / `ml/training/` read.

2. **Counterparty Knowledge Base — extracted, but NOT integrated, NOT consumed by
   anything.** `ml/labeling/knowledge_base/counterparty_knowledge.csv` (406 rows,
   generated by `build_counterparty_knowledge.py`), classifying each recipient pattern
   as `family`/`self`/`friend`/`merchant`/`subscription_service`/`medical_provider`/
   `fee_or_debt_service`/`person_unconfirmed`/`unknown`. This preserves the "who is
   this" reasoning that would otherwise only exist as prose in `LABELING_STATUS.md` —
   but it is a standalone data file, not wired into the app, database schema, or ML
   pipeline in any way. Read `counterparty_knowledge.README.md` before using it for
   anything, especially the confidence column: only 32 of 406 rows are individually
   confirmed relationships (`high` confidence); the other 374 are either mechanically
   derived from the spending category or from a bulk "classify remaining names as
   friends" delegation the user made to speed up labeling, never confirmed per-name.

Treat "build a queryable, production-consumable counterparty knowledge base" as
unstarted future work (`docs/roadmap.md` Phase 9) — `counterparty_knowledge.csv` is raw
material for that future work, not a finished feature.

**What Epic 4 should actually train on:** `ml/data/spendwise_labeled.xlsx`'s `category`
column only — exactly the 12 values in `docs/requirements.md`. Nothing else.
